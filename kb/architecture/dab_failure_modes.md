DataAgentBench: Four Hard Failure Categories
Here is how this works.
DataAgentBench (DAB) was designed by UC Berkeley's EPIC Data Lab specifically to expose failure modes that do not appear in clean single-table benchmarks. Every DAB query tests at least one of four hard requirements. Agents that fail on DAB almost always fail in one of these four ways.
Failure Category 1 — Multi-database routing failure. A single query requires data from two or more database systems (e.g., PostgreSQL and MongoDB in the same answer). The agent must route sub-queries to the correct database, translate between query dialects (SQL vs. MongoDB aggregation pipeline vs. DuckDB analytical SQL), and merge results without data loss. Agents fail here by querying only one database, or by attempting a cross-database join that the execution engine cannot run.
Failure Category 2 — Ill-formatted join key mismatch. Entity references are formatted differently across databases. A customer ID that is an integer (12345) in a PostgreSQL transaction table may appear as "CUST-00123" in a MongoDB CRM collection. The agent must detect the mismatch and resolve it before attempting the join — without being told explicitly. Agents fail here by attempting the join on the raw values and returning zero results or wrong matches.
Failure Category 3 — Unstructured text extraction failure. Some queries require extracting structured facts from free-text fields before those facts can be used in a calculation. For example, counting "negative sentiment mentions" in support notes requires an extraction step before counting. Agents fail here by returning raw text instead of a structured count, or by skipping the extraction step entirely.
Failure Category 4 — Domain knowledge gap. Answering correctly requires knowing something not present in the schema. The term "active customer" may seem unambiguous, but the correct definition in a given domain is "purchased in the last 90 days," not simply "has a row in the customer table." Agents fail here by using a naive proxy (row existence) when the correct definition is domain-specific.
The adversarial probe library must cover at least three of these four categories, with 15+ probes total.

Injection test question: An agent returns zero results when joining customer transactions to CRM records. Which DAB failure category is this, and what is the likely root cause?
Expected answer: Failure Category 2 — ill-formatted join key mismatch. The customer ID is likely formatted differently across the two databases (e.g., integer vs. prefixed string), causing the join to find no matching rows.
